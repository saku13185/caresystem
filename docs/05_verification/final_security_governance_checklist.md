# 최종 보안 및 거버넌스 체크리스트 (Final Security & Governance Checklist)

본 문서는 스마트시티 안심 돌봄 관리 에이전트 시스템을 실제 프로덕션 환경에 실증 배포하기 전, 시스템의 안전성, 신뢰성 및 규제 준수 여부를 확인하기 위해 수립된 **최종 보안 및 AI 거버넌스 체크리스트**입니다. 

기존에 해결된 기술 부채와 검증 결과를 기반으로 실제로 **확인 완료된 내용(Verified)**과 향후 운영을 위한 **권고사항(Recommended)**을 엄격히 구분하여 명세합니다.

## 1. 🔑 접근 제어 체크리스트 (Access Control)

사회복지사가 접근하는 웹 대시보드의 인증 신뢰성과 노인 데이터 보호를 위한 가드 동작을 검증합니다.

| 검증 항목 | 분류 | 상세 확인 내용 및 가이드라인 | 상태 |
| :--- | :--- | :--- | :--- |
| **마스터 패스워드 장벽** | 확인 완료 | `MASTER_PASSWORD_HASH` 환경 변수가 누락된 경우 로그인 폼 대신 경고를 표출하며 `st.stop()`으로 화면 진입을 즉각 차단함. | **통과 (Verified)** |
| **단방향 암호화 해싱** | 확인 완료 | 패스워드를 평문으로 보존하지 않고, PBKDF2-SHA256 해시 포맷(`pbkdf2_sha256$<iterations>$<salt>$<hash_hex>`)으로 저장 및 검증함. | **통과 (Verified)** |
| **타이밍 공격 차단** | 확인 완료 | 해시 대조 시 `hmac.compare_digest` 연산을 수행하여 시간 차 공격(Timing Attack)에 의한 패스워드 유출을 원천 방지함. | **통과 (Verified)** |
| **세션 보안 유지** | 확인 완료 | 브라우저별 세션이 완전히 격리되며, 비밀번호 원문을 `st.session_state`나 로그에 노출/저장하지 않음. | **통과 (Verified)** |
| **로그아웃 기능** | 확인 완료 | 사이드바 하단에 "로그아웃" 기능을 제공하여 인증 세션을 즉각 파기시킬 수 있음. | **통과 (Verified)** |
| **다중 계정 및 권한 제어** | 권고 사항 | 상용 실증 배포 시, 단일 마스터 패스워드 체계를 탈피하고 복지사별 개별 ID/PW를 발급받는 OAuth2/OIDC 토큰 인증 도입을 권장함. | **권고 (Recommended)** |

| 검증 항목 | 분류 | 상세 확인 내용 및 가이드라인 | 상태 |
| :--- | :--- | :--- | :--- |
| **비식별화 식별자 작동** | 확인 완료 | 노인의 이름, 실주소, 주민번호를 직접 수집하지 않고 UUID v4 및 비식별화 가상 코드(`RES-MASK-2026A` 등)로만 데이터를 매핑함. | **통과 (Verified)** |
| **PII 검출 및 FP 차단** | 확인 완료 | 검증 도구(`check_db.py`) 고도화를 통해 이름, 주민번호, 이메일, 전화번호 누출을 실시간 감지함. 일반 명사(이탈, 이상)를 이름으로 판정하는 오탐(False Positive) 로직 차단 완료. | **통과 (Verified)** |
| **하네스 자동화 테스트** | 확인 완료 | `test_pii_detection.py`에 구현된 21건의 PII 단독/문맥 누출 검출 테스트 시나리오가 모두 정상 작동함. | **통과 (Verified)** |
| **HTTPS 전송 암호화** | 권고 사항 | 실증 배포 웹 서버 기동 시 브라우저 통신 전 구간에 TLS 1.3 암호화를 적용하여 스니핑 공격에 의한 데이터 탈취를 차단해야 함. | **권고 (Recommended)** |

## 9. 🚧 남은 한계 (Remaining Limitations)

1. **단일 로그인 인증 가드**: 본 대시보드 보안은 하나의 마스터 비밀번호 SHA-256 해시를 공유하여 사용하므로 복지사 개개인별 권한 격리 및 사후 조치 책임성 추적에 계정적 한계가 존재합니다.
2. **GenAI Live API 레이턴시 지연**: 외부 네트워크 연동 시 5~15초 수준의 응답 지연이 발생할 수 있으나, 일간 요약 배치 시스템의 한계로 수용 범위 내에 있습니다.
3. **Docker CLI 환경 제약**: WSL 커널 호환성 이슈로 인해 에이전트 내 직접 배포판 기동 테스트는 불가하여 최종 컨테이너 구동은 호스트 실측 장비의 검증 결과에 의존합니다.

## 10. 🎯 다음 단계 권고 (Next Steps Recommendations)

1. **[보안 레벨업] HTTPS + OAuth2 연동**: 실 배포 전 웹 안심 도메인을 바인딩하고 사내 인증 허브(SSO, OAuth2)를 대시보드 세션 가드에 연계하여 개인정보 유출 리스크를 근본적으로 해소할 것을 권장합니다.
2. **[성능 보완] 훈련 Tensor 제로 카피 최적화 (`TECH-DEBT-04`)**: 차기 스프린트 마일스톤으로 모델 학습 과정에서 발생하는 메모리 복사 레이턴시를 최소화하기 위해 훈련 코드 내 `torch.tensor` 복사를 `torch.from_numpy` 포인터 제로 카피 연산으로 변경하는 최적화를 진행할 것을 제안합니다.
3. **[감사 기능 고도화] 감사 로그 기록**: `caregiver_alerts` 조치 데이터 처리와 관련하여 감사 목적의 불변 기록 테이블(Audit Logs)을 데이터베이스 스키마에 추가하고 복지사의 조치 기록을 누적할 것을 권장합니다.
