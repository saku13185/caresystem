## 1. システム概要

### 1-1. システム名と目的

**システム名**: 独居高齢者 安心予防ケア AI エージェントシステム（Care Agent System）

**目的**:  
スマートホームに設置されたセンサーから取得した「日常生活活動（ADL: Activities of Daily Living）」の41種類の行動占有率データを分析し、以下の処理を自動で行うシステムです。

1. **AttentionRNN 深層学習モデル** で翌日の行動パターンを予測
2. **Double-step 異常検知スコアリング** で異常パターンを早期発見
3. **GenAI（Gemini/OpenAI）XAI 説明可能性レポート** を自動生成し、社会福祉士の迅速な対応を支援

### 1-2. 技術スタック

| 分類 | 技術 |
|------|------|
| 言語 | Python 3.10+ |
| 深層学習 | PyTorch（GRU + Self-Attention） |
| データベース | SQLite（WAL モード） |
| ダッシュボード | Streamlit（ダークモード・グラスモーフィズム） |
| グラフ描画 | Plotly |
| LLM 連携 | Google Gemini API（google-genai SDK）/ OpenAI API |
| テスト | pytest（BDD シナリオベース） |
| コンテナ | Docker / Docker Compose |

## 2. アーキテクチャ設計

### 2-1. レイヤー構成（DDD + クリーンアーキテクチャ）

```
domain/ ← usecases/ ← infrastructure/ ← presentation/
```

依存方向は一方向のみ（上流→下流への依存は禁止）。

| レイヤー | パス | 役割 |
|---------|------|------|
| ドメイン | `src/domain/` | 純粋なビジネスルール・エンティティ（Resident, Anomaly, Alert） |
| ユースケース | `src/usecases/` | ビジネスロジック（前処理・異常検知オーケストレーション） |
| インフラ | `src/infrastructure/` | DB・モデル・スコアラー・LLM の具体実装 |
| プレゼンテーション | `src/presentation/` | Streamlit ダッシュボード・認証ガード |

### 2-2. 依存性逆転（DIP）の実装

ユースケース層は具体的なDB実装に依存しません。  
抽象ポートインターフェース `CareRepositoryPort` を介して疎結合を実現しています。

```
usecases/ports/care_repository.py  ← 抽象インターフェース（Protocol）
infrastructure/persistence/sqlite_care_repository.py  ← 具体実装アダプター
infrastructure/persistence/db_connector.py  ← SQLite 接続コア
```

## 4. 主要コンポーネント詳細

### 4-1. AttentionRNN モデル（`src/infrastructure/models/attention_rnn.py`）

**アーキテクチャ**: Self-Attention 付き GRU（Gated Recurrent Unit）ニューラルネットワーク

**入出力仕様**:
- 入力: 過去15日間の ADL 点有率データ `(Batch, 15, 41)`
- 出力①: 翌日（16日目）の予測点有率 `(Batch, 41)` 
- 出力②: 各日の貢献度を示す Attention 重み `(Batch, 15)`

**モデル構成**:
```
GRU 層       : input_dim=41, hidden_dim=64, batch_first=True
Attention FC : Linear(64, 1) → Softmax(dim=1) で各日の重要度スコア算出
Output FC    : Linear(64, 41) で翌日の41活動占有率を予測
```

**学習済み重みファイル**: `attention_rnn.pt`（プロジェクトルートに配置）

---

### 4-2. AttentionRNN モデルキャッシュ（`src/infrastructure/models/model_cache.py`）

複数入居者を一括分析する際に重みファイルをディスクから毎回ロードする非効率を解消します。

- `(model_path, device)` キーでインメモリ辞書にキャッシュ
- `threading.Lock()` でスレッドセーフを保証
- 最初の1回のみ `torch.load()` を実行、2回目以降はキャッシュから取得

**確認できるログ出力例**:
```
[AttentionRNNModelCache] Loading model from disk: attention_rnn.pt (cpu)   ← 初回のみ
[AttentionRNNModelCache] Reusing cached AttentionRNN model for key: ...    ← 2回目以降
```

---

### 4-3. Double-step 異常スコアラー（`src/infrastructure/scorers/double_step_scorer.py`）

**1段階目: MAE Z-score 検証**（全体的な行動パターン変化の検出）
- 予測値と実測値の平均絶対誤差（MAE）を計算
- `Z-score = (MAE - 過去MAE平均) / 過去MAE標準偏差`

**2段階目: Boxplot IQR 検証**（個別活動の局所的異常の検出）
- 41活動それぞれについて、歴史的分布の IQR 範囲外（Q1 - 1.5×IQR ～ Q3 + 1.5×IQR）を逸脱しているか確認

**最終危険度判定基準**:

| 判定 | 条件 |
|------|------|
| `DANGER`（高リスク・2） | Z-score > 2.5 かつ Sleep/Eat の Boxplot Low 逸脱あり |
| `WARNING`（注意・1） | Z-score > 1.5 または Boxplot 逸脱が1件以上 |
| `NORMAL`（正常・0） | 上記に該当しない |

---

### 4-4. XAI レポート生成（`src/infrastructure/llm/xai_report_generator.py`）

Google Gemini API（google-genai SDK）を使用して、異常検知結果を社会福祉士向けの自然言語レポートに変換します。

**Fallback 設計**（API 障害時のシステム継続保証）:
1. `GEMINI_API_KEY` が未設定・ダミー値の場合 → ルールベース日本語代替テキストを即時生成
2. API 通信中に Exception 発生 → `try-except` で安全に隔離しログ出力後、Fallback を再帰呼び出し
3. API キーはログ出力時に正規表現でマスク処理（`AIzaSy***[MASKED_GEMINI_KEY]***`）

**医療免責表示**（ハードコーディング）:
```

予防及びモニタリング補助目的のみに使用してください。
```

## 9. 処理フロー（実行時のデータフロー）

```
[毎日バッチ実行]
          │
          ▼
RunAnomalyDetectionUseCase.execute(resident_id, target_date)
          │
          ├─① CareRepositoryPort.get_resident()          → DB から入居者情報取得
          │
          ├─② PreprocessADLDataUseCase.execute()          → 過去15日ウィンドウ構築
          │                                                  （欠損値補完含む）
          ├─③ AttentionRNNModelCache.get_model()          → 重み読込（初回のみ）
          │     └─ AttentionRNN.forward(x_window)          → 予測値 + Attention 重み
          │
          ├─④ CareRepositoryPort.get_daily_adl_summaries() → 歴史的分布データ取得
          │
          ├─⑤ DoubleStepScorer.compute_mae_z_score()      → Z-score 計算
          │     DoubleStepScorer.check_boxplot_violations() → IQR 逸脱チェック
          │     DoubleStepScorer.determine_danger_level()   → 危険度判定
          │
          ├─⑥ XaiReportGenerator.generate_report()        → Gemini API / Fallback
          │     XaiReportGenerator.generate_caregiver_memo() → 推奨メモ生成
          │
          └─⑦ DB 永続化:
                ├─ update_resident_status()   → residents テーブル更新
                ├─ insert_anomaly_report()    → anomaly_reports テーブル挿入
                └─ insert_caregiver_alert()   → caregiver_alerts テーブル挿入
```

## 10. 既知の問題と対処法

### 問題1: `seed_data.py` 実行時にエラーログが大量出力される

```
[XAI_GENERATOR_ERROR] Google GenAI API Exception: ...
```

**原因**: `.env` に `GEMINI_API_KEY` が未設定またはダミー値  
**対処**: これはシステムの設計された Fallback 機構が動作中の**正常動作**です。ダッシュボードを起動するとルールベースの代替テキストが正常に表示されます。実際の AI 分析レポートが必要な場合は有効な API キーを `.env` に設定してください。

### 問題2: Windows PowerShell で `Activate.ps1` 実行時に権限エラー

**対処**: PowerShell を管理者権限で開き、以下を実行:
```powershell
Set-ExecutionPolicy RemoteSigned
```

### 問題3: Docker コンテナで SQLite の `database is locked` エラー

**原因**: Windows NTFS ホストパスへの bind mount による POSIX ファイルロック不一致  
**対処**: `docker-compose.yml` の named volume（`care_data`）構成を使用することで解決済み。bind mount は使用しないでください。

### 問題4: `seed_data.py` で `NameError: name 'os' is not defined`

**対処**: `seed_data.py` の先頭に `import os` が含まれる最新バージョンを使用してください。

## 14. 連絡先・引き継ぎ事項

- 本システムは研究・プロトタイプ段階のソフトウェアです。本番環境への導入前に、セキュリティ監査と倫理審査を実施することを強く推奨します。
- `.env` ファイルは Git 管理外のため、引き継ぎ時は別途安全な経路で共有してください。
- `care_system.db` はシードデータが投入済みです。テスト用データが含まれています。
- `attention_rnn.pt` は学習済みモデル重みです。モデルを再学習する場合は `src/infrastructure/models/train_pipeline.py` を実行してください。
