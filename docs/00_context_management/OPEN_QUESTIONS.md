## 7. 구현 과정에서 도출된 기술 부채 및 리팩토링 코드 스멜 (Technical Debt & Code Smell)

### [TECH-DEBT-01] 딥러닝 추론 시 모델 가중치 파일의 중복 디스크 I/O 및 스레드 안전성
* **상세 내용**: `RunAnomalyDetectionUseCase` 실행 시 `ModelTrainer`를 통해 `.pt` 모델 가중치를 디스크에서 매번 동적으로 동기식 로드하여 추론하고 있음. 대량의 피돌봄 노인 데이터를 자정 일괄 배치 처리 시, 매 노인마다 가중치 파일을 반복해서 읽어 들이는 불필요한 디스크 I/O 레이턴시가 누적될 우려가 있음.
* **해결 방안**: 대시보드 구동 시 혹은 배치 프로세스 초기화 시 모델을 단 1회만 메모리에 캐싱(Singleton/In-memory Cache)하여 스레드 안전하게 다중 노인 데이터를 고속 분석하도록 아키텍처 리팩토링이 필요함.

### [TECH-DEBT-02] 다차원 다변량 NDArray와 PyTorch Tensor 간 형변환 오버헤드 및 정밀도
* **상세 내용**: 전처리 레이어에서는 `numpy.float32`를, PyTorch AttentionRNN 모듈에서는 `torch.float32`를, 통계 Scorer 레이어에서는 `numpy.float64` 및 파이썬 원시 `float` 타입을 빈번하게 크로스 캐스팅하고 있음. 연산 오버헤드와 정밀도 손실 위험이 잔존함.
* **해결 방안**: 입력부터 최종 통계 연산까지 데이터 타입을 정합성 있게 고정(Unification)하고, 캐스팅 오버헤드를 통제하는 타입 린팅 검사를 도입할 필요가 있음.

### [TECH-DEBT-03] 유스케이스 레이어의 구체 인프라(DB Connector) 직접 의존에 따른 DIP 위반
* **상세 내용**: `src/usecases/` 내부 흐름 제어 로직이 `src/infrastructure/`에 속하는 구체적 SQLite 데이터베이스 클래스를 직접 임포트하여 강결합(Tight Coupling)된 스멜이 잔존함.
* **해결 방안**: 2차 리팩토링 단계에서 유스케이스 레이어 내부에 추상 포트 인터페이스(IADLRepository)를 정의하고 의존성 주입(DIP)을 강제하도록 격리함.

### [TECH-DEBT-04] PyTorch 텐서 변환 시 메모리 복사 발생에 따른 성능 오버헤드
* **상세 내용**: 학습 파이프라인에서 numpy 배열을 PyTorch 텐서로 변환 시 `torch.tensor()`를 호출하여 불필요한 메모리 복사 및 캐시 지연이 유발됨.
* **해결 방안**: 주소 공유 및 오버헤드 제로인 `torch.from_numpy()` 방식으로의 핫 텐서 리팩토링을 수행함.

### [TECH-DEBT-05] Streamlit 대시보드 내 다중 세션 관리자 인증 및 인가 가드 레이어 부재
* **상세 내용**: 사회복지사용 웹 대시보드가 인증(Auth) 없이 구동되어 스마트시티 외부망 노출 시 비식별 상태의 독거노인 생활 패턴 및 XAI 리포트가 임의 노출될 리스크가 있음.
* **해결 방안**: 2차 상용 배포 전 전용 로그인 컴포넌트(마스터 키 또는 OAuth2.0 연동) 세션을 대시보드 상단 가드 레이어로 강제 결합할 계획 수립.

### [TECH-DEBT-06] 예외 캐치 로그 출력 시 민감 외부 키(sk-...) 노출 가능성
* **상세 내용**: LLM API 호출 예외 포착 시 에러 트레이스백(`str(e)`)을 무가공 출력하고 있어, 스택 출력에 결합된 일부 API 키 정보나 민감 파라미터가 디버그 파일 로그 상에 노출될 소지가 있음.
* **해결 방안**: 예외 로깅 모듈 전반에 정규식 기반 API 키 포맷 마스킹 필터를 인터셉터 형태로 적용하여 로그 평문 노출을 영구 격리 차단함.
