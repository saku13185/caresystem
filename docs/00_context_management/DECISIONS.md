# 주요 기술 및 비즈니스 의사결정 이력 (DECISIONS)

본 문서는 프로젝트의 핵심 아키텍처 및 기획적 의사결정 사항과 그 범위(In-Scope / Non-Scope)를 추적하기 위해 작성되었습니다.

---

## 1. 코어 서비스 범위 (In-Scope)

### [DECISION-01] CASAS 규격 기반 시계열 합성 데이터 제너레이터 구축
* **내용**: 실제 물리 센서 부재 문제를 해결하기 위해, CASAS 표준 데이터셋을 모사한 41개 핵심 일상생활 행동(ADL)의 일간 시간 점유율(%) 합성 데이터를 생성한다.
* **이유**: 실데이터 획득의 한계를 돌파하고 모델 학습 및 검증 환경을 안정적으로 확보하기 위함이다.

### [DECISION-02] AttentionRNN 기반 16일 차 행동 예측 모델 채택
* **내용**: 과거 15일간의 행동 윈도우 데이터를 입력받아 다음 날(16일 차)의 활동 패턴 점유율을 예측하는 Attention-based RNN 아키텍처를 모델의 중추로 채택한다.
* **이유**: 단순 시계열 예측을 넘어 어떠한 날의 활동 정보가 예측에 주요하게 기여했는지 어텐션 가중치(Attention Weights)를 통해 해석할 수 있는 기반을 다지기 위함이다.

### [DECISION-03] Double-step 이상 탐지 파이프라인 구축
* **내용**: 1차적으로 실제 점유율과 예측 점유율 간의 MAE를 계산한 후 Z-score로 환산하고, 2차적으로 각 활동별 역사적 데이터 분포의 Boxplot 임계치를 상회하는지 대조하는 이중 검증(Double-step) 탐지 로직을 수행한다.
* **이유**: 이상 패턴 감지의 신뢰성을 확보하고 오탐율을 최소화하기 위함이다.

### [DECISION-04] LLM 기반 설명 가능성(XAI) 자연어 리포트 생성
* **내용**: Z-score 분류 결과와 편차가 큰 활동 정보(Context)를 LLM 프롬프트에 주입하여, 사회복지사가 직관적으로 이해할 수 있는 한글 돌봄 리포트를 자동 생성한다.
* **이유**: 비전문가인 돌봄 인력이 복잡한 시계열 수치를 해석하지 않고도 신속하고 정확한 의사결정을 내릴 수 있도록 돕기 위함이다.

## 2. 명시적 제외 범위 및 MVP 아키텍처 제약조건 (Non-Scope & MVP Freeze Constraints)

### [NON-SCOPE-01] [동결(Freeze)] 물리 IoT 센서 연동 및 실시간 하드웨어 수집 배제
* **내용**: 스마트홈 물리 센서(Zigbee, Z-Wave, BLE 등) 하드웨어 드라이버, 게이트웨이 통신 모듈 및 실시간 소켓 수집 서버를 절대 구현하지 않는다.
* **제약**: 데이터 수집은 오직 CASAS 스키마를 준수하는 '합성 데이터 제너레이터'의 파일 입출력 배치 동작으로 제한한다.

### [NON-SCOPE-02] [동결(Freeze)] 임상 의료 진단 및 최종 의료 처방 배제
* **내용**: AI 리포트가 질병 명칭을 확정 진단하거나 약물 처방 등의 의료 행위로 오인될 수 있는 표현 방식을 철저히 차단한다.
* **제약**: 보고서 생성 엔진의 템플릿 프롬프트 및 UI 컴포넌트 상단에 "의사결정 보조지표" 경고 필터 문구를 하드코딩하여 이를 강제 제약한다.

### [NON-SCOPE-03] [동결(Freeze)] 초단위/분단위 실시간 스트리밍 분석 배제
* **내용**: 실시간 스트리밍 처리를 위한 분산 이벤트 브로커(Kafka 등) 및 스트림 연산 프레임워크 설계는 완전히 배제한다.
* **제약**: 모든 시계열 예측과 이상치 분석은 일 단위 배치(Batch) 파일 프로세싱 모델로 단일 아키텍처를 동결한다.

### [NON-SCOPE-04] [동결(Freeze)] 고도화된 계정 권한 및 모바일 앱 이식 배제
* **내용**: OAuth2.0 소셜 로그인, JWT/MFA 권한 검증 및 네이티브 모바일 애플리케이션(iOS/Android) 빌드/패키징 시도를 철저히 배제한다.
* **제약**: 인증 레이어는 격리된 단일 Mock 복지사 계정 세션으로 단순화하며, 사용자 UI는 Streamlit 단일 반응형 웹 인터페이스로만 동결 구현한다.

## 3. 사용자 역할(Role) 및 인가 범위 (Role & Authorization Boundaries)

### [ROLE-01] 독거노인 (Primary Actor)의 비활성(Passive) 인가
* **규칙**: 독거노인 개인은 시스템에 직접 로그인하거나 데이터를 조회/조작할 수 없다. 
* **설계 제약**: 프라이버시 보호를 위해, 센서 이벤트 기반의 일상 데이터 수집 시 노인의 직접적인 개인식별정보(PII)는 배제하고 난수화된 행동 식별자(예: UUID)와 점유율 수치 형태로만 변환되어 저장 및 가공된다.

### [ROLE-02] 사회복지사 및 생활지원사 (Core User)의 상위 의사결정 인가
* **규칙**: 사회복지사는 관내 독거노인 전체의 위험도 등급(정상/주의/고위험) 및 자연어 XAI 분석 리포트를 조회할 수 있는 전적인 권한을 갖는다.
* **설계 제약**: 사회복지사는 이상 탐지 결과에 대한 오탐(False Positive) 여부를 피드백할 수 있으며, 보호자에게 비상 위험 상황 알림을 최종 전송/승인하는 관문(Gateway) 역할을 수행한다.

### [ROLE-03] 보호자 (Core User)의 제한적 조회 인가
* **규칙**: 보호자는 원천 ADL 데이터나 비식별 해제된 실시간 원시 로그에 직접 접근할 수 없으며, 오직 사회복지사가 확인 및 승인한 노인의 '안심 요약 및 정제된 위험 경보 리포트'만 조회가 가능하다.
* **설계 제약**: 보호자의 인가는 본인이 등록한 독거노인 1인에 대해서만 제한적으로 매핑되어 정보 조회가 강제 격리된다.

### [ROLE-04] 시스템 에이전트 간 약결합(Loosely Coupled) 및 데이터 보안
* **규칙**: AttentionRNN, Double-step 이상 분류기, LLM 리포트 추론기 등의 소프트웨어 에이전트는 노인의 개인정보를 직접 처리할 수 없으며, 난수화된 익명 ID(UUID)와 계량화된 활동 데이터 패킷만을 교환하여 보고서를 구성해야 한다.
* **설계 제약**: LLM 프롬프트 생성 시에는 노인의 실명이나 정확한 주소 등은 격리하고, 예방 조치 시 필요한 비식별화된 패턴 지표만을 제공한다.

## 4. 코어 요구사항 ID 목록 (Core Requirements ID Matrix)

### [FR-MATRIX] 기능 요구사항 ID 목록
* **FR-01**: CASAS 규격 모사 시계열 합성 데이터 제너레이터 (41개 ADL 일간 시간 점유율 % 생성)
* **FR-02**: AttentionRNN 기반 시계열 예측 엔진 (과거 15일 윈도우 인풋, Self-Attention 가중치 추출)
* **FR-03**: Double-step 이상 탐지 및 Z-score 분류기 (정상, 주의, 고위험 3단계 위험 분류)
* **FR-04**: LLM XAI 자연어 리포트 생성기 (한국어 보고서 자동 완성 및 환각 제어)

### [NFR-MATRIX] 비기능 요구사항 ID 목록
* **NFR-01**: 성능 및 추론 시간 제한 (AttentionRNN 추론 < 100ms, LLM XAI 생성 < 5s(상용)/15s(로컬))
* **NFR-02**: 데이터 무결성 및 보정 (일간 점유율 합산 오차 $\pm 0.01\%$ 정밀 정규화 보장 및 결측치 Imputation 적용)
* **NFR-03**: 윤리적 제약 및 보안 정책 (의료 오인 방지 경고문 하드코딩 표출, PII 완전 차단 및 UUID v4 대체)

## 5. 도메인 애그리거트 경계 및 비즈니스 용어 확정 (DDD Aggregates & Ubiquitous Language)

### [AGGREGATES] 핵심 도메인 애그리거트 (Aggregate Roots & Entities)
* **Resident 애그리거트 (Root: Resident)**:
  - *Resident (Entity)*: 독거노인의 비식별 UUID 기반 계정 정보 제어.
  - *DailyADLSummary (Entity)*: 일일 자정 집계되는 41개 일상 활동의 시간 점유비 요약 정보 수집.
  - *SensorEvent (Entity)*: 합성 데이터 생성기로부터 생성된 가상 로우 레벨 이벤트 로그.
* **Analysis 애그리거트 (Root: AnomalyReport)**:
  - *AnomalyReport (Entity)*: 예측 MAE의 Z-score 산출 수치, RNN Attention 15일 가중치 배열, 그리고 한글 자연어 돌봄 설명 리포트 통합.
  - *BoxplotViolation (Value Object)*: 41개 개별 행동의 박스플롯 임계 이탈 상세 정보 격리 (불변 밸류 객체).
* **Alert 애그리거트 (Root: CaregiverAlert)**:
  - *CaregiverAlert (Entity)*: 사회복지사 대시보드 내 대기/승인/반려 의사결정 기록, 복지사 메모 피드백 보존 및 보호자 경보 전송 인가.

### [UBIQUITOUS-LANGUAGE] 비즈니스 용어 사전 (Ubiquitous Language)
* **ADL Share (일상생활 활동 점유비)**: 24시간 동안 노인이 행한 41개 핵심 ADL(수면, 식사, 위생 등) 각각의 정규화된 시간 백분율(%) 수치.
* **Double-step Anomaly Check (이중 이상 탐지)**: 1단계 예측 오차의 Z-score 검증(전체 일상 왜곡 측정)과 2단계 개별 활동 Boxplot IQR 검증(국소 아웃라이어 측정)을 결합한 이중 통계 검증 프로세스.
* **Context Injection XAI (맥락 주입형 설명 가능성 AI)**: Z-score 수치, Boxplot 이탈 및 Attention 기여 정보를 LLM 프롬프트에 context로 자동 주입하여 한글 보고서를 동적 도출하는 엔진.
* **Caregiver Feedback Gateway (실무자 피드백 관문)**: 사회복지사가 분석 정보를 확인하고 정탐(승인/경보 전송)/오탐(반려) 피드백을 기록하여 데이터베이스에 격리 등록하는 경보 시스템 관문.

## 6. 아키텍처 패턴 및 기술 스택 확정 (Architecture Patterns & Tech Stack)

### [ARCH-PATTERN] 클린/헥사고날 아키텍처 패턴 채택
* **내용**: 밖에서 안으로 의존성이 흐르는 클린 아키텍처 아키텍처 구조화 규정을 채택한다.
* **이유**: 신경망 라이브러리(PyTorch) 및 웹 UI(Streamlit), 파일 보존 기술 등의 인프라 구성 요소를 핵심 비즈니스(도메인) 및 유스케이스로부터 논리 격리시켜, 외부 컴포넌트 교체 시의 도메인 파급 오류를 완벽 차단하기 위함이다.

### [ARCH-DEPENDENCY] 레이어 간 의존성 방향의 엄격화
* **내용**: `domain` -> `usecases` -> `infrastructure` / `presentation` 방향으로만 의존성(참조)을 허용한다.
* **이유**: `domain` 레이어는 프레임워크나 외부 벤더 통신 API를 절대 알 수 없어야 하며, `usecases`는 흐름 포트(Interface/Abstract class)만을 제공해 `infrastructure`에서 상세 모듈을 구현하게 함으로써 테스트 용이성 및 결합도를 낮추기 위함이다.

### [TECH-STACK] MVP 구현 기술 세부 사항
* **데이터 과학 및 딥러닝**: `PyTorch`, `NumPy`, `SciPy` (Z-score 및 IQR 통계 분석 연산 수행)
* **LLM XAI 및 프롬프팅**: `OpenAI Python SDK` 또는 로컬 API 어댑터
* **사용자 표현 화면**: `Streamlit` (복지사 안심 대시보드 구현)
* **파일 영속화 저장소**: `Python Built-in File I/O` 및 `CSV` 포맷팅 (로컬 연구 자립 배포 보장)

## 7. 시계열 데이터 저장 구조 및 보존 정책 (Time-Series Data Storage & Retention Policy)

### [DATA-STORAGE-STRATEGY] 정규 관계형 물리 모델 및 로컬 SQLite 채택
* **내용**: 파일 기반 영속화(CSV)와 SQL 질의 정합성을 동시에 만족하기 위해, 표준 RDB 물리 모델 스키마를 준수하며 로컬 샌드박스 구동에 특화된 `SQLite` 임베디드 파일 데이터베이스 파일(`.db`) 형태로 영속성 데이터를 우선 관리 운용한다.
* **이유**: 향후 대규모 서버용 PostgreSQL 등 다른 RDB로 마이그레이션이 수월하며, 복잡한 다차원 복합 인덱스를 빠르게 적용하기 위함이다.

### [DATA-RETENTION-POLICY] 시계열 테이블별 수명주기(TTL) 동결
* **원천 센서 이벤트 로그 (`casas_sensor_events`)**: 
  - *정책*: `daily_adl_summaries`로 일별 전처리 및 정규화 합산($100.00\%$) 완료가 검증 통과된 시점 기준 **최근 30일치만 보존**하고, 30일을 초과하는 가상 원천 로그는 매일 자정 자동 퍼지(Purge) 삭제한다.
  - *이유*: 수천만 건에 달할 수 있는 원천 이벤트 데이터 용량 폭증을 차단하고 개인 프라이버시 노출 면적을 원천 격리 축소하기 위함이다.
* **일간 점유율 요약 데이터 (`daily_adl_summaries`)**: 
  - *정책*: AttentionRNN 모델의 장기 패턴 재학습(Re-training) 및 노인 생활 주기의 다년도 변이 트렌드 추적을 위해 **최소 3개년(1095일) 동안 영구 보존**한다.
* **이상 보고서 및 비상 알림 이력 (`anomaly_reports`, `caregiver_alerts`)**: 
  - *정책*: 고위험 감지 이력 및 사회복지사의 정오탐 조치 피드백 메모는 돌봄 서비스의 법적 책임 추적성 및 서비스 감사 신뢰도 확보를 위해 **완전 무기한(영구) 보존**한다.

## 8. 개발 시퀀스 및 스프린트 백로그 마일스톤 (Development Sequence & Backlog Milestones)

## 9. 데이터베이스 마이그레이션 전략 (Database Migration Strategy)

### [DECISION-05] 연구 및 서비스 단계별 데이터베이스 마이그레이션 전략
* **내용**: MVP 단계의 로컬 `SQLite` 환경에서 향후 실제 상용 운영 단계인 `PostgreSQL`로의 데이터 및 스키마 이전을 지원하기 위해 다음과 같은 마이그레이션 아키텍처 및 전략을 도입한다.
  1. **스키마 버전 관리**: 가볍고 유연한 스키마 진화를 지원하기 위해, `db_connector.py` 내부 또는 SQLite 메타 테이블(`schema_versions`)을 사용하여 현재 버전을 추적하고 DDL 변경분을 순차 실행하는 버전 제어 체계를 사용한다.
  2. **ORM 및 쿼리 이식성**: SQL 데이터 조작 시 SQLite 고유 방언을 배제하고, 표준 ANSI SQL(ISO/IEC 9075)만을 사용하거나 의존성 격리 포트/어댑터 패턴을 철저히 준수하여 비즈니스 로직 수정 없이 데이터베이스 커넥터 클래스만 교체할 수 있도록 설계한다.
  3. **데이터 이관 파이프라인**: 로컬 SQLite 파일(`.db`)의 비식별화 데이터(UUID v4 기반)를 PostgreSQL 서버로 벌크 인서트(Bulk Insert)할 수 있는 Python 마이그레이션 마스터 스크립트 도구를 제공하여, 노인의 개인정보 유출 리스크 없이 가상 행동 시계열 데이터를 백업 및 복구할 수 있게 한다.
  4. **호환 가능한 인덱스 이식**: SQLite 수준에서 정의된 `idx_sensor_events_resident_time`, `idx_adl_summary_resident_date` 등의 복합 인덱스는 PostgreSQL의 B-Tree 인덱스 스펙으로 1:1 매핑 이식되어 데이터 용량이 1억 건 이상으로 증가해도 10ms 이내의 윈도우 조회 쿼리 성능을 보장한다.
* **이유**: 연구실 환경의 단일 로컬 실행 수준을 넘어 스마트시티 중앙 분석 서버로의 유연한 도메인 확장 및 스케일아웃을 사전에 안정적으로 확보하기 위함이다.

## 10. AttentionRNN 핵심 하이퍼파라미터 및 손실 함수 규격 명세 (Model Hyperparameters & Loss Function Specification)

### [DECISION-06] AttentionRNN 예측 신경망 아키텍처 파라미터 및 손실 함수 규격 확정
* **내용**: 15일 슬라이딩 윈도우 기반 16일 차 다변량 예측을 수행하는 `AttentionRNN` 모델의 핵심 하이퍼파라미터 및 훈련 환경 사양을 아래와 같이 동결한다.
  1. **슬라이딩 입력 윈도우 크기 (Window Size)**: 과거 **15일** 고정 (`(Batch, 15, 41)`)
  2. **입력 및 출력 변수 피처 수 (Feature Dimensions)**: CASAS 41개 ADL 점유비 고정 (`41`차원)
  3. **GRU 은닉 상태 차원 (Hidden Dimensions)**: **64**차원
  4. **최적화 도구 (Optimizer)**: **Adam** (기초 학습률: `0.005`)
  5. **손실 함수 (Loss Function)**: **L1 Loss (Mean Absolute Error, MAE)**
     - *채택 이유*: 예측과 실측 간의 MAE 오차 자체가 이상 유무를 판정하는 1단계 Z-score의 기초 통계량으로 직접 사용되므로, 훈련 손실 함수와 판정 지표 간의 수학적 일관성(Mathematical Alignment)을 확보하기 위해 L1 Loss를 채택함.
  6. **Self-Attention 가중치 규격**: 시간 스텝(15일) 차원에 대해 Softmax 함수를 취하여 합이 정확히 `1.0`이 되는 `(Batch, 15)` 크기의 어텐션 가중치 맵 산출.
* **이유**: 통계적 분류 모형의 수학적 일관성을 확보하고 기하급수적으로 왜곡될 수 있는 복잡성을 방지하여 AI의 예측 설명 가능성을 극대화하기 위함이다.

## 11. 프론트엔드 시각화 라이브러리 및 상태 관리 설계 (Frontend Visual Library & State Management Design)

### [DECISION-07] Streamlit & Plotly 기반 웹 시각화 레이아웃 및 상태 동기화 확정
* **내용**: 사회복지사 전용 예방 돌봄 대시보드의 화면 구성, 차트 라이브러리 및 피드백 상태 제어 흐름을 다음과 같이 최종 동결한다.
  1. **화면 프레임워크**: **Streamlit (단일 웹 애플리케이션 반응형 웹)**
     - *채택 이유*: 데이터 분석 및 연구 개발 결과물을 복잡한 웹 빌드 인프라 없이 단시간 내에 고성능 데이터 프로토타입 인터페이스로 배포하기 위함임.
  2. **차트 라이브러리**: **Plotly Graph Objects (Interactive go.Bar)**
     - *채택 이유*: 사용자가 마우스를 올릴 때 동적으로 기여 가중치 및 임계 편차 백분율을 호버 정보로 명확히 표시하여, 정적 이미지 차트 대비 데이터 탐색성과 전문성을 극대화하기 위함임.
  3. **디자인 테마**: **HSL 기반 다크 모드 글라스모피즘 (Glassmorphism)**
     - *채택 이유*: Plain RGB 원색의 경보 알림을 배제하고, 눈의 피로도를 낮추는 HSL 기반의 세련된 다크 테마와 세련된 UI 카드를 렌더링하여 실무 복지사에게 Wow Factor와 함께 고도화된 안심 감각을 제공하기 위함임.
  4. **상태 관리 및 피드백 동기화 흐름**: 
     - 복지사가 정탐(APPROVED)/오탐(REJECTED)을 클릭하고 조치 메시지를 저장하는 순간, SQLite DB의 `caregiver_alerts` 및 `residents` 테이블의 상태값을 실시간으로 SQL 트랜잭션 업데이트 수행.
     - 저장 완료 즉시 `st.rerun()` 호출을 유도해 화면 상의 대상자 카드 리스트 및 위험도 배지를 실시간 비동기 동기화 렌더링함.
* **이유**: 단순 목업 수준을 넘어 실제 실무 환경에서 복지사가 AI 분석 결과를 신뢰성 있게 수용 및 모니터링하고 피드백할 수 있도록 인지 공학적 인터페이스 신뢰성을 확보하기 위함이다.

## 12. 개인정보(PII) 난독화 비식별화 및 보안 통제 설계 (PII Anonymization & Security Control Design)

### [DECISION-08] 무조건적 PII 완전 제거형 UUID v4 및 마스킹 연구 코드 확정
* **내용**: 독거노인의 프라이버시 보호법(개인정보보호법 제15조) 준수를 위해 다음 보안 난독화 설계 정책을 코어 시스템 엔진 수준에서 영구 동결한다.
  1. **실데이터 격리화**: 데이터베이스 스키마 및 파일 시스템 내에 노인의 성명, 주소, 연락처, 주민등록번호 등 어떠한 직접적인 개인식별정보(PII) 컬럼 설계도 원천 배제한다.
  2. **가상 비식별 키 체계**: 노인의 고유 키는 오직 `UUID v4` 난수 체계로 통제하며, 대시보드 화면 및 리포트에 표출되는 식별자는 난독화 마스킹 코딩된 가상 연구 코드(예: `RES-MASK-2026A`) 형태로만 변환되어 격리 표출된다.
  3. **LLM 외부 전송 격리**: 설명 가능성(XAI) 분석을 위해 OpenAI API 또는 외부 AI API 서버로 프롬프트를 전송 시, 오직 Z-score 수치, Boxplot 이탈 및 UUID 기반 가상 코드 정보만을 전송하며 어떠한 원천 민감 정보도 포함하지 않는다.
* **이유**: 정보 유출 사고 시에도 노인의 프라이버시가 침해당할 소지를 수학적·아키텍처적으로 원천 격리 차단하기 위함이다.
