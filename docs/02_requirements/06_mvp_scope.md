# MVP 개발 범위 정의서 (MVP Scope Planning)

본 문서는 기획 및 요구사항 명세 단계에서 도출된 다양한 요구사항을 바탕으로, 제한된 일정 내에 시스템의 핵심 가치를 성공적으로 검증하기 위해 MoSCoW 방법론을 적용하여 초기 구현 범위를 명확히 정의하고 동결(Freeze)하기 위해 작성되었습니다.

---

## 1. MoSCoW 스코프 매트릭스

| 우선순위 | 핵심 기능 및 구성 요소 | 비고 |
| :--- | :--- | :--- |
| **Must-Have**<br>(초기 구현 필수) | * CASAS 규격 41개 ADL 일간 시간 점유율 합성 데이터 제너레이터<br>* AttentionRNN 훈련 및 추론 시계열 딥러닝 파이프라인<br>* Double-step 이상 탐지 분류기 (Z-score + Boxplot IQR 검증)<br>* LLM XAI 한글 돌봄 리포트 생성기 (개인정보 격리 및 의료 진단 배제 경고 필터)<br>* Streamlit 기반 복지사 대시보드 프로토타입 UI (분석 결과 표출 & 피드백 창구) | **MVP 개발의 코어 범위** |

## 2. MVP 필수 구현 범위 (Must-Have & Should-Have Detail)

### 2.1. 시계열 합성 데이터 제너레이터
* **범위**: CASAS 표준 스키마를 모사한 41개 ADL 일간 시간 점유율(%) 합성 데이터를 생성합니다. 
* **MVP 제약**: 하루 총 활동 점유율 합은 반드시 **$100.00\%$ ($\pm 0.01\%$ 오차 보강)**을 보장하는 정규화 함수가 포함되어야 하며, 파라미터 제어를 통해 불면증 시나리오 및 식사 거부 등의 질병 변이를 강제 주입하여 모델을 훈련/검증할 수 있어야 합니다.

### 2.2. AttentionRNN 훈련 및 추론 파이프라인
* **범위**: 과거 15일간의 41차원 다변량 점유율 윈도우 데이터를 바탕으로 16일 차를 예측하고 학습 기여 가중치를 산출하는 모델을 구축합니다.
* **MVP 제약**: 인풋 형태는 `(Batch, 15, 41)`, 아웃풋 예측 형태는 `(Batch, 41)`로 엄격히 통제하며, Self-Attention 가중치 배열 `(Batch, 15)`을 추출하는 API 레이어를 반드시 제공해야 합니다.

### 2.3. Double-step 이상 분류기
* **범위**: 예측치와 실측치 간 MAE의 Z-score 산정(1단계) 및 개별 이상 활동의 역사적 IQR Boxplot 아웃라이어 판정(2단계)을 동적 수행합니다.
* **MVP 제약**: 정상(0), 주의(1), 고위험(2)의 3단계 등급 판정을 내리는 완전 자동화 코드로 구축합니다.

### 2.4. LLM XAI 자연어 보고서 생성기
* **범위**: Z-score, Boxplot 위반 항목, Attention 가중치를 입력 컨텍스트로 LLM에 주입하여 신뢰도 높은 한글 돌봄 리포트를 출력합니다.
* **MVP 제약**: 보고서 최상단에 **"의료적 최종 판단 아님"을 밝히는 보조 지표 안내 문구**를 강제 하드코딩 표출하고, 개인정보 유출 방지를 위해 노인의 인적 사항은 UUID v4로 완전히 격리 대체합니다.

### 2.5. Streamlit 기반 복지사 대시보드 UI
* **범위**: 위 파이프라인의 분석 결과를 한눈에 모니터링할 수 있는 경량 웹 대시보드를 구축합니다.
* **MVP 기능**:
  - 위험 등급별(정상/주의/고위험) 대상 노인 리스트 보드.
  - 예측 편차 및 15일 Attention 가중치 분포 시각화 차트.
  - LLM XAI 한글 예방 리포트 팝업 카드.
  - 해당 리포트 판정에 대한 사회복지사의 '정탐/오탐 피드백' 및 메모 등록 창구.

## 3. 스코프 부활 방지 가드레일 (Scope Guardrails)

개발 진행 중 스코프 비대화(Scope Creep)로 인한 프로젝트 지연을 막기 위해 아래의 강력한 차단 가드레일을 선언하고 아키텍처 의사결정으로 강제 규정합니다.

1. **[물리 센서 드라이버 구현 금지]**:
   - Raspberry Pi, Zigbee, Z-Wave 등 하드웨어 연동을 위한 통신 드라이버, 실시간 소켓 서버, 또는 센서 물리 인터페이스를 구현하는 모든 코드는 백엔드 코드베이스에서 엄격히 금지됩니다. 데이터 수집 단계는 오직 '합성 데이터 제너레이터' 파일 입출력으로 한정합니다.
2. **[고도화된 보안/인증 레이어 구현 제한]**:
   - JWT, OAuth 2.0, 다중 MFA, Spring Security 등 복잡한 다중 관리자 권한 및 계정 관리 로직을 구축하지 않습니다. MVP 프로토타입 단계에서는 단순 격리된 단일 사회복지사 세션(Mock User)으로 보안 영역을 단순화합니다.
3. **[네이티브 배포 및 모바일 이식 금지]**:
   - React Native, Flutter, Swift/Kotlin 연동 등을 통해 스마트폰 애플리케이션으로의 빌드 및 패키징 시도를 아키텍처 수준에서 강력히 차단합니다. 복지사 대시보드는 **Streamlit 단일 반응형 웹 인터페이스**로 한정 구동합니다.
4. **[실시간 스트리밍 분석 배제]**:
   - 초단위로 데이터가 쏟아지는 Kafka, Spark Streaming 등 분산 메시지 큐와 스트리밍 분석 프레임워크의 도안 및 설계 시도를 철저히 배제하고, 배치(Batch) 파일 파싱 모델로 아키텍처를 고정합니다.
